# FlashInbox / 闪收箱 Spec

## 1. 背景与目标

### 1.1 背景

用户希望快速获得一个可用邮箱地址，用于注册/验证码/通知接收。服务部署在 Cloudflare Serverless（Pages/Workers），数据库使用 D1，前后端均用 React（Next.js）实现，UI 风格为 Material Design 3（MDUI）。

### 1.2 产品目标

1. **即刻可用**：无需注册账号即可生成邮箱并查看收件箱。
2. **可恢复访问**：通过 `username + key`（密钥租约）在不同设备恢复查看邮件。
3. **邮件不丢（在策略窗口内）**：任何投递到受管域名的邮件都被存储，即使邮箱未创建。
4. **强防串号**：确保创建时不撞库，且“key 恢复”不会指向错误邮箱。
5. **可运营**：具备管理员后台、域名管理、规则（丢弃/隔离）与数据看板。
6. **安全合规**：HTML 邮件安全渲染、限流、反爆破、隐私提示明确。

------

## 2. 非目标（明确不做 / MVP 不包含）

- 不提供 IMAP/POP3；不做传统“邮箱客户端协议”。
- 不存储任何附件（二进制内容全部丢弃）。
- 不提供“完整新建外发邮件”的通用 SMTP 发信箱能力；仅定义可选“回复（reply）”。
- 不做用户账号体系（用户名/密码/多角色注册系统）。
- 不承诺无限期保存邮件（保留策略由产品定义）。

------

## 3. 平台依赖与硬约束

### 3.1 入站邮件

使用 Cloudflare Email Routing + Email Workers 来接收并处理入站邮件。Email Routing 当前不支持大于 **25 MiB** 的邮件。([Cloudflare Docs](https://developers.cloudflare.com/email-routing/limits/?utm_source=chatgpt.com))

### 3.2 回复能力（可选）

如实现“往返”中的回复：Email Workers 支持对入站邮件进行 reply，但存在关键限制（如同一 EmailMessage 事件只能 reply 一次、收件人与发件域约束等），需要在产品层明确。([Cloudflare Docs](https://developers.cloudflare.com/changelog/2025-03-12-reply-limits/?utm_source=chatgpt.com))

### 3.3 Next.js 部署

Next.js 推荐通过 OpenNext adapter 部署到 Cloudflare Workers（全栈同构）或前端用 Pages、后端用 Workers 分离。([Cloudflare Docs](https://developers.cloudflare.com/workers/framework-guides/web-apps/nextjs/?utm_source=chatgpt.com))

## 4. 术语与定义

- **domain**：受管邮箱域名，如 `mail.example.com`
- **username / local-part**：邮箱前缀，如 `BluePanda23`
- **邮箱地址**：`username@domain`
- **邮件（message）**：入站邮件解析后的可展示内容（不含附件）
- **未认领邮箱（Unclaimed mailbox）**：收到了邮件但尚未被用户“创建/认领”的地址
- **已认领邮箱（Claimed mailbox）**：用户通过创建或认领获得访问权的地址
- **key（密钥）**：邮箱访问授权的“租约凭证”，**有效期 15 天**，可续期
- **续期（renew）**：在 key 未过期且验证成功的情况下延长 key 有效期
- **销毁（destroy）**：key 过期未续期时，访问权失效并按策略清理数据

## 5. 用户角色与权限

### 5.1 访客用户（匿名）

- 可随机生成邮箱
- 可手动指定前缀创建邮箱（受规则限制）
- 可查看当前会话下的邮箱收件箱（通过会话凭证）
- 可对“未认领邮箱”发起认领（受强风控限制）

### 5.2 保留用户（持有 key）

- 可使用 `username + key (+domain)` 恢复访问
- 可续期 key（延长 15 天）
- 可查看该邮箱在保留窗口内的所有邮件

### 5.3 管理员（持 ADMIN_TOKEN）

- 进入管理面板
- 管理域名（增删、启用、禁用、只读）
- 管理规则（丢弃/隔离）
- 查看隔离队列并释放/删除
- 查看审计日志与数据看板

## 6. 产品核心机制与安全模型

### 6.1 “不撞库”与防串号（关键原则）

**原则 P1：`username@domain` 一旦被系统识别出现过（创建/认领/收到邮件），则永不再次分配给不同 mailbox 身份。**

> 这会增加命名空间消耗，但能彻底规避“旧 key + username 看到了新邮箱”的串号风险。

**原则 P2：key 永远绑定到唯一邮箱身份（mailbox identity），且 key 过期后必须不可恢复（除非再次认领生成新 key）。**

### 6.2 未创建先收信：查看必须先认领（Claim）

为解决你提出的矛盾：“未创建邮箱也收到了邮件，但查看需要 key”，系统引入**认领流程**：

- 当任意 `username@domain` 收到邮件时，即使邮箱未创建，系统也会创建/维护一个**未认领邮箱记录**并存储邮件。
- 用户若想查看该邮箱的邮件：
  - 若邮箱 **已认领**：必须 `username + key` 验证通过才能查看
  - 若邮箱 **未认领**：必须先通过 **Claim（认领）**，系统生成 key（只展示一次），从此该邮箱转为已认领；后续查看一律需要 key

> **认领是“获得访问权”的关键操作**，因此必须有更强反滥用门槛（见第 12 章）。

### 6.3 Key 的租约语义（15 天过期 + 续期 + 自动销毁）

- key 是“访问授权租约”，有效期为 **15 天**
- 续期只能在 key 未过期且验证成功时进行
- 过期未续期：自动销毁访问授权；并触发数据清理策略（见 11.4）

## 7. 功能需求总览（MVP 必须实现）

1. 生成邮箱：随机 / 手动指定前缀
2. 收信：存储邮件元信息、纯文本正文、净化后的 HTML（不存附件）
3. 查看：收件箱列表、邮件详情、安全 HTML 渲染
4. 认领（Claim）：对未认领邮箱生成 key 并转为已认领
5. 恢复（Recover）：`username + key` 恢复访问
6. 续期（Renew）：延长 key 有效期
7. 管理后台：token 登录、域名管理、规则管理、隔离队列、数据看板
8. 反滥用：按 IP/ASN/UA 限流（创建/认领/恢复分级）；恢复入口限流更严格
9. 内容安全：HTML 禁脚本、外链默认不加载
10. 丢弃/隔离：按发件人域/地址/关键词/IP（best-effort）规则执行
11. 隐私提示：明确存储邮件内容、不建议接收敏感信息

## 8. 邮箱命名与创建规则

### 8.1 随机生成 username 规则（必须满足其一）

- 规则 RND-1：驼峰命名英文名（CamelCase），例如 `BluePanda`
- 规则 RND-2：驼峰命名英文名 + 2–6 位数字，例如 `BluePanda23`、`BluePanda204811`

约束建议：

- 总长度建议上限（例如 ≤ 24）以保证可读性
- 使用词库组合（形容词/动物/自然名词等），需过滤敏感词（脏词表）

### 8.2 手动指定 username 规则

- 字符集：英文字母 + 数字（可选允许 `_`/`-`，但建议默认禁用以降低歧义）
- 长度：建议 3–30
- 规范化：用于判重的 canonical 形式必须一致（如统一小写 + 去首尾空格）

### 8.3 唯一性与并发（硬性）

- 同一 domain 下，username 必须全局唯一
- 创建流程必须具备并发安全：不会出现两个用户同时获得同一地址

### 8.4 手动前缀的安全限制（强烈建议）

由于手动前缀可预测性更强：

- **未创建但收到了邮件**：默认进入隔离/不可认领区（仅管理员可见）
- 仅随机生成的地址允许“未创建先收信后认领”的路径（降低猜测攻击面）

------

## 9. 邮箱状态机（Mailbox State Machine）

定义 mailbox 在系统中的生命周期状态：

1. **UNCLAIMED（未认领）**
   - 触发：邮箱收到邮件但用户尚未创建/认领；或 key 过期后按策略回落
   - 可见性：普通用户不可见邮件内容（可显示“该地址存在邮件但未认领”这类占位信息，但不泄露发件人/主题等元数据，避免侧信道）
2. **CLAIMED（已认领）**
   - 触发：用户创建并选择保留，或用户对 UNCLAIMED 执行 Claim 成功
   - 特征：必须 key 才能查看
3. **ACTIVE_SESSION（会话可访问）**（不是主状态，是访问态）
   - 用户当前设备持有会话凭证，可免重复输入 key（在会话有效期内）
4. **QUARANTINED（隔离）**（按邮件粒度，但对邮箱层可显示“隔离计数”）
   - 命中规则的邮件进入隔离队列，用户不可见，管理员可审

> 说明：邮箱“创建但不保留”的模式如果你仍保留，可以作为 CLAIMED 的一种子类型（但因你强调 key 过期销毁，推荐所有可查看路径最终都落到 key 上）。

## 10. Key 状态机与租约生命周期

### 10.1 Key 状态

- **VALID（有效）**：可用于恢复与查看
- **EXPIRED（过期）**：不可用于恢复/查看
- **REVOKED（撤销）**（可选增强）：管理员或用户主动撤销

### 10.2 过期策略（本 Spec 采用强销毁策略）

- key 在 15 天到期未续期：
  - 访问授权立即失效
  - 系统执行**自动销毁**：删除邮箱下所有可读邮件内容（或整封邮件记录），并将 mailbox 回落为 UNCLAIMED/DESTROYED（实现可二选一）
  - 保留最小审计信息（如邮箱标识、销毁时间、原因）用于反滥用追踪

> 这样最符合“自动销毁”的直觉，也避免“过期后再次认领拿到历史邮件”的风险。

### 10.3 续期（Renew）规则

- 仅当 `username + key` 校验成功且 key 未过期时允许续期
- 续期后过期时间重置为“当前时间 + 15 天”
- 对续期也需要限流（比查看弱、比恢复略强）

## 11. 邮件接收、解析、存储与展示

### 11.1 入站处理（Email Worker 逻辑需求）

每封到达受管域名的邮件：

1. 解析收件人（To/Envelope）映射到 `username@domain`
2. 若 mailbox 不存在：创建 UNCLAIMED mailbox（或创建占位记录）
3. 解析并存储邮件的必要字段：
   - 基本头：From、To、Subject、Date、Message-ID
   - 线程头：In-Reply-To / References（如用于线程）
   - MIME 摘要：是否含 text/plain、text/html、附件数量（仅计数）
4. 内容存储：
   - text/plain：保存（可截断）
   - text/html：保存“净化版本”（可截断）
5. 附件处理（见 11.3）
6. 丢弃/隔离规则判断（见第 13 章）

### 11.2 邮件大小与截断策略

- 平台限制：Email Routing 不支持 >25 MiB 邮件，超过则接收失败或无法投递，需在运维日志中可见。([Cloudflare Docs](https://developers.cloudflare.com/email-routing/limits/?utm_source=chatgpt.com))
- 系统内部：对正文（plain/html）必须设定最大存储与展示上限（避免 D1 单行限制、避免 UI 卡死）
- UI 必须明确标记“内容已截断”

### 11.3 附件：不存储任何附件（硬性）

- 系统不存储附件二进制内容
- 可选：存储附件元信息（文件名/MIME/大小/哈希），用于提示“该邮件含附件但已丢弃”
- UI 必须明确提示：**附件不保存，无法下载**

### 11.4 数据保留与销毁

- 关键策略：**key 过期未续期 = 自动销毁**
- “自动销毁”至少包含：
  - 删除该 mailbox 下所有可读内容（建议连同邮件记录一起删除或做不可逆匿名化）
  - 保留最小审计项（用于反滥用与统计）
- 对 UNCLAIMED mailbox（未认领）也可设定最大保留期（例如 7 天），超期自动清理（可作为运营参数）

### 11.5 HTML 安全渲染（必须）

要求：**禁用脚本、外链资源默认不加载**。

- 净化规则（实现无关）：
  - 移除所有脚本/iframe/object/embed 等高风险标签
  - 移除所有事件属性（onload/onclick…）
  - 阻断危险 URL scheme（javascript:, data: 中的可执行内容等）
- 外链资源：
  - 外链图片/样式默认不加载（替换占位）
  - 提供“加载外部资源”按钮（需二次确认并提示风险）
- 展示优先级：
  - 默认展示纯文本（如存在）
  - 用户可切换到“HTML 视图”（展示净化后的 HTML）

## 12. 访问控制与反滥用（Rate Limit / Bot / Brute Force）

### 12.1 限流维度（硬性）

对以下动作都必须限流：

- 创建邮箱（Create）
- 认领邮箱（Claim）
- 恢复访问（Recover）
- 续期（Renew）
- 管理后台登录与敏感操作（Admin）

限流键必须综合：

- **IP**
- **ASN**
- **UA 哈希（UA fingerprint 的一部分）**

### 12.2 限流分级（推荐强度顺序）

**Recover（最严） > Claim（很严） > Create（中等） > Read（较宽松）**

建议行为：

- 触发限流时返回冷却时间（Retry-After 概念）
- Claim/Recover 触发限流后应强制 Turnstile 人机验证（或更严格挑战）

### 12.3 认领（Claim）的强风控要求（必须）

Claim 的目标是“首次获得未认领邮箱的访问权”，因此必须：

- 强限流（按 IP/ASN/UA）
- 人机验证（强烈建议 Turnstile）
- 可选 PoW（客户端计算挑战）抵抗批量脚本
- 不泄露元数据：Claim 前不得展示该邮箱任何邮件主题/发件人/时间等信息

### 12.4 恢复入口防爆破（必须）

- `username + key` 校验失败需：
  - 计数并限流（更严格）
  - 可引入逐步惩罚：冷却时间指数增长
- 对比校验需要恒定时间比较（降低侧信道）

------

## 13. 丢弃 / 隔离规则系统（Drop / Quarantine）

### 13.1 规则类型（必须支持）

- 发件人域：`*@bad.com`
- 发件人地址：`bad@xxx.com`
- 关键词：主题/正文匹配（支持多关键词）
- 发件 IP（若可获得）：从邮件头 best-effort 提取（不保证可靠）

### 13.2 规则动作

- **DROP（丢弃）**：不保存内容，仅保留最小审计（时间、原因、收件地址、发件域/地址）
- **QUARANTINE（隔离）**：保存但对用户不可见，进入管理员隔离队列
- **ALLOW（放行）**：进入正常入库与展示流程

### 13.3 规则执行顺序（必须定义并保持一致）

建议顺序：

1. 明确允许（白名单，若你做）
2. DROP
3. QUARANTINE
4. 默认 ALLOW

### 13.4 隔离队列（管理员功能）

- 查看隔离邮件（内容可见）
- 释放（转入正常邮箱）
- 永久删除
- 规则命中解释（此邮件命中了哪些规则）

------

## 14. 域名管理与配置策略（D1 + Env）

### 14.1 配置分层（硬性）

- **环境变量（Secrets / Vars）**：敏感与部署级配置，不能暴露在代码/前端包中
  - `ADMIN_TOKEN`
  - `DEFAULT_DOMAIN`
  - 其他安全参数（例如 pepper、限流阈值默认值等）
- **D1 数据库**：运行期可变配置
  - 域名列表：启用/禁用/只读/备注
  - 规则列表：丢弃/隔离规则
  - 仪表盘聚合数据（可选）

### 14.2 域名的三态（必须）

- **Enabled**：允许前台创建与认领
- **Disabled**：不允许创建/认领（但可选择是否仍接收并存储到 UNCLAIMED）
- **ReadOnly**：不允许创建/认领，但允许接收（适合迁移期或风险域）

### 14.3 Cloudflare Email Routing 依赖说明（必须在后台提示）

应用层“启用域名”≠ Cloudflare 侧已完成路由配置。后台必须提示管理员：

- 该域名需要在 Cloudflare Email Routing 完成接收配置，否则邮件无法到达 Worker（或无法按预期到达）。

------

## 15. 管理后台（Admin Console）

### 15.1 认证（硬性）

- 管理员通过 `ADMIN_TOKEN` 进入后台
- Token 只在服务端校验，**不得出现在 URL 参数中**，不得写入前端 bundle
- 登录成功后生成 **admin session**（短期会话），用于后续操作授权

### 15.2 URL 追踪参数（你要求必须有）

后台访问 URL 自动附加参数用于审计追踪（不包含敏感信息）：

- `ts`：时间戳
- `fp`：浏览器弱指纹（建议为若干非敏感字段组合哈希：UA、timezone、language、platform、screen）
- `sid`：admin_session_id（随机、短期）

注意事项（必须写进 Spec）：

- URL 参数可能被历史记录、复制粘贴、Referrer 泄露，所以只用于追踪，不用于鉴权关键凭证

### 15.3 后台信息架构（必须）

- 仪表盘（Dashboard）
- 域名管理（Domains）
- 规则管理（Rules：drop/quarantine）
- 隔离队列（Quarantine）
- 审计日志（Audit Logs）
- 系统设置（Settings，仅展示非敏感配置；敏感配置仅提示“已配置/未配置”）

### 15.4 数据看板指标（必须）

至少包含：

- 邮件接收量（按小时/天）
- UNCLAIMED/CLAIMED 数量
- Create/Claim/Recover/Renew 请求量与成功率
- Recover 失败分布（限流/校验失败/过期）
- 丢弃/隔离命中数（按规则类型）
- HTML 净化触发计数（剥离脚本/阻止外链等）
- 存储与解析错误计数（写入失败、解析失败、超限）

------

## 16. 前端 UI/UX（Next.js + MDUI/MD3）

### 16.1 页面清单（必须）

1. 首页（Create）
   - 随机生成邮箱（主 CTA）
   - 手动指定前缀（次 CTA）

## 19. 关键流程（端到端）

### 19.1 随机创建邮箱（Create Random）

1. 用户选择域名（默认 DEFAULT_DOMAIN）
2. 点击“随机生成”
3. 系统生成 username 候选并保证唯一
4. 创建成功：
   - 返回邮箱地址
   - 建立访问会话（会话可短期免 key）
   - 用户若选择“立即认领/保留”：生成 key（15 天）并展示一次

### 19.2 手动创建邮箱（Create Manual）

1. 用户输入 username
2. 前端做格式校验（不含唯一性）
3. 服务端校验唯一性并创建
4. 若选择保留：生成 key 并展示一次

### 19.3 邮件入站到未创建邮箱

1. 邮件到达 `x@domain`
2. 系统发现 mailbox 不存在：
   - 创建 UNCLAIMED mailbox（或占位记录）
   - 存储邮件（不含附件）
3. 普通用户不可查看内容，除非后续 Claim/创建并转为 CLAIMED

### 19.4 认领（Claim）未认领邮箱（必须强风控）

1. 用户输入邮箱地址并通过人机验证
2. 系统检查该地址是否允许被认领（建议仅限随机风格地址）
3. 认领成功：
   - mailbox -> CLAIMED
   - 生成 key（15 天）展示一次
4. 之后查看一律需要 `username + key`

### 19.5 恢复（Recover）与续期（Renew）

1. 用户输入 `username + key (+domain)`
2. 校验成功：
   - 签发会话（短期免重复输入）
   - 提供续期按钮
3. 续期成功：
   - key 过期时间重置为现在 + 15 天

### 19.6 key 过期自动销毁

1. 定期检查 key 是否过期
2. 对过期且未续期的邮箱执行销毁策略：
   - 删除邮件内容
   - mailbox 回落（或标记 DESTROYED）
   - 保留最小审计项

## 20. 验收标准（Acceptance Criteria）

### 20.1 创建与唯一性

- 并发创建不会出现同一 `username@domain` 被发放给两个用户
- 已出现过的地址不会被再次分配给其他邮箱身份（防串号）

### 20.2 未创建先收信

- 投递到未创建地址的邮件会被存储
- 在未认领前，用户无法查看任何内容或元数据
- 认领后可查看历史邮件（在保留窗口内）

### 20.3 key 15 天与销毁

- key 在 15 天后失效
- 失效后无法恢复查看
- 自动销毁按策略执行（内容不可再访问）

### 20.4 HTML 安全

- HTML 邮件中脚本不会执行
- 外链资源默认不加载，需用户手动确认

### 20.5 反滥用

- Create/Claim/Recover 都有按 IP/ASN/UA 的限流
- Recover 失败会触发更严格限流与冷却
- Claim 必须有人机验证（或等价强挑战）

### 20.6 管理后台

- 仅凭 ADMIN_TOKEN 可进入后台
- URL 自动追加 ts/fp/sid
- 后台具备域名管理、规则管理、隔离队列、审计日志、数据看板

## 23. Umami 追踪支持（Analytics）与安全约束

### 23.1 目标

- 支持接入 **Umami** 做站点行为追踪（页面浏览、基础事件），用于运营分析与反滥用趋势监控。
- 必须做到：**不引入脚本注入/XSS**，不因动态拼接导致第三方脚本被劫持；并保证管理后台与用户侧埋点隔离（最小必要原则）。

### 23.2 范围

**必须支持：**

- PV（页面访问）统计
- 可选事件：点击“随机生成”、点击“认领”、Recover 成功/失败（仅记录结果类别，不记录输入）
- 分环境开关（dev/staging/prod）
- 管理后台也可统计，但与用户站点数据区分（不同 website id 或不同 instance）

**不做：**

- 录屏、热力图
- 采集用户输入内容、邮件内容、key、邮箱地址明文等任何敏感信息

### 23.3 配置与部署

- Umami endpoint、website id 必须通过 **环境变量** 注入（不能写死在前端代码仓库里作为常量，避免误暴露配置与跨环境污染）。
- 支持两种模式（择一或同时支持）：
  1. **自托管 Umami**（推荐）：由你控制域名与脚本来源，便于 CSP 收紧与隐私合规
  2. **第三方 Umami 服务**：必须通过 CSP 白名单严格限定脚本源与上报域

> 说明：website id 本身通常不算敏感，但仍建议走 env，便于分环境与快速切换；真正敏感的是任何管理密钥与后台 token（本 Spec 已要求仅 env 且服务端校验）。

### 23.4 防注入 / XSS 安全要求（硬性）

为避免 Umami 相关引入注入风险，必须满足：

1. **脚本来源固定且不可由用户输入影响**

- Umami script URL、data-website-id 等参数只能来自“受信任配置源”（env + 服务端渲染注入），**禁止**从 query、localStorage、邮件内容、任何用户可控字段拼接。

1. **禁止内联脚本注入**

- 不允许在页面中拼接 `<script>${string}</script>` 或使用 `dangerouslySetInnerHTML` 注入任何 Umami 相关内容。
- 若必须插入 `<script>` 标签，只能使用框架提供的安全机制（例如 Next.js 的 Script 组件）并且属性值必须来自受信任配置。

1. **CSP（Content-Security-Policy）约束（强烈建议并作为验收项）**

- `script-src` 必须只允许：
  - `'self'`
  - Umami script 域名（若第三方）
  - 禁止 `unsafe-inline`（除非你使用 nonce 且严格控制）
- `connect-src` 必须只允许：
  - `'self'`
  - Umami 上报 endpoint
- 管理后台可单独更严格的 CSP（尤其禁止任何外域脚本，除 Umami 必需项）

1. **Referrer Policy**

- 由于管理后台 URL 会附加 `ts/fp/sid` 参数，为防止被外链泄露：
  - 对管理后台页面必须设置 `Referrer-Policy: no-referrer` 或 `strict-origin`（推荐 no-referrer）。

1. **事件埋点数据最小化**（硬性）

- 事件 payload 禁止包含：
  - 邮箱地址、username、key、邮件主题、发件人、正文、任何可能识别个人的信息
- 可记录的字段（允许）：
  - action 名称（固定枚举）
  - 成功/失败布尔与失败原因类别（固定枚举，如 rate_limited / invalid_key / expired）
  - 环境（env）、页面路径（path）

### 23.5 事件命名规范（建议）

- `create_random_click`
- `create_manual_submit`
- `claim_submit`
- `recover_submit`
- `recover_success`
- `recover_fail_rate_limited`
- `recover_fail_invalid`
- `renew_success`
- `admin_login_success`
- `admin_login_fail`

### 23.6 验收标准（Umami）

- 禁止出现可被用户输入影响的 Umami script URL / endpoint
- CSP 生效：页面无 `unsafe-inline`（或使用 nonce 且验证）
- 管理后台访问外链时不会携带包含 `sid` 的 referrer
- 事件上报不包含邮箱、key、邮件内容等敏感字段

## 附录 A：邮箱/邮件不可泄露元数据清单（Claim 前）

> 目标：避免攻击者通过“试探某地址是否存在/是否有邮件/邮件来自谁/何时收到”等信息构建侧信道，从而定位到“有价值的地址”再去爆破 Claim/Recover。

### A.1 Claim 前（UNCLAIMED 状态）对普通用户必须不可见的内容

对任意未认领邮箱地址，在未通过 Claim 或合法会话前，**一律不可返回**：

**邮箱存在性与活跃性**

- 该 `username@domain` 是否存在记录
- 是否收到过邮件（有/无）
- 收到邮件数量（哪怕是 1）
- 最后一封邮件时间、第一封邮件时间、任何时间范围
- 未读/已读状态（根本不应存在）
- 邮箱是否被创建过、是否被销毁过、是否曾被认领

**邮件列表相关**

- 邮件主题（Subject）
- 发件人地址/发件人域（From）
- 收件人（To/CC/BCC）字段
- 邮件日期（Date）
- Message-ID / thread 信息
- 邮件大小、MIME 类型、是否含 HTML、是否含附件（即使你不存附件也不能泄露“含附件”这一事实）
- 邮件 snippet / preview / 正文任何片段

**策略与规则相关**

- 是否命中丢弃/隔离规则
- 命中了哪条规则、命中原因
- 隔离队列里是否存在针对该地址的邮件

**系统行为差异（侧信道）**

- 响应时间差异：对“存在/不存在/有邮件/无邮件”的情况，响应体与耗时应尽量一致
- 错误信息差异：不得出现 “邮箱不存在/邮箱无邮件/邮箱已销毁” 等可枚举提示

### A.2 Claim 前允许展示的“最小信息”

- **统一提示文案**（固定）：例如“该地址需要认领后才能查看”
- **固定动作**：展示 Claim 的入口（但不回显任何邮箱状态）
- **通用限流错误**：例如“请求过于频繁，请稍后再试”（不可暗示地址价值）

### A.3 Claim 后（CLAIMED）对持 key 用户允许可见的内容

Claim 成功后，才允许访问：

- 邮件列表（主题/发件人/时间/状态）
- 邮件详情（正文、净化 HTML）
- 隔离邮件仍不可见（除非你决定“隔离对用户可见但标记”，本 Spec 默认不可见）

## 附录 B：限流阈值建议表（按 IP / ASN / UA）

> 说明：这是“建议默认值”，需可通过环境变量或后台策略参数调整。
> 限流键：`action + ip + asn + ua_hash`（组合键降低误伤）。
> 建议采用“固定窗口 + 冷却”或“滑动窗口 + 指数退避”均可，但对 Recover/Claim 必须支持更严格惩罚。

### B.1 动作分级与阈值（建议默认）

| 动作                                    | 风险等级 | 建议阈值（组合键） | 触发后的惩罚                      | 备注                           |
| --------------------------------------- | -------- | ------------------ | --------------------------------- | ------------------------------ |
| Create（随机/手动）                     | 中       | 10 次 / 10 分钟    | 冷却 2 分钟；重复触发升至 10 分钟 | 允许正常使用，但防批量刷号     |
| Claim（认领）                           | 高       | 3 次 / 30 分钟     | 冷却 30 分钟；强制人机验证        | Claim 能解锁未知地址，必须更严 |
| Recover（恢复）                         | 最高     | 5 次 / 60 分钟     | 冷却 60 分钟；失败指数退避        | 防爆破 key                     |
| Renew（续期）                           | 中高     | 10 次 / 60 分钟    | 冷却 10 分钟                      | 防自动续期滥用                 |
| Read Inbox/List                         | 低中     | 120 次 / 10 分钟   | 429 + 短冷却                      | 防抓取与压测                   |
| Read Message Detail                     | 中       | 60 次 / 10 分钟    | 429 + 短冷却                      | 防批量下载内容                 |
| Admin Login                             | 高       | 5 次 / 60 分钟     | 冷却 120 分钟                     | 管理端更敏感                   |
| Admin Mutations（改域名/规则/释放隔离） | 高       | 60 次 / 10 分钟    | 冷却 10 分钟；记录审计            | 防误操作/脚本化                |

### B.2 Recover 失败的指数退避（建议）

当同一组合键下 Recover 连续失败：

- 第 1–2 次：仅记录

- 第 3 次：冷却 5 分钟

- 第 4 次：冷却 15 分钟

- 第 5 次：冷却 60 分钟

- > 5 次：冷却 24 小时 + 强制人机验证

> 同时建议在“key 错误”与“key 过期”上统一模糊提示，避免攻击者区分有效/无效 key。

### B.3 高风险网络加严（建议规则）

对常见机房/代理 ASN（可维护名单）：

- Create 阈值减半
- Claim/Recover 阈值进一步减半
- 直接要求 Turnstile（即使未触发限流）

### B.4 验收标准（限流）

- Create/Claim/Recover 均按 IP/ASN/UA 生效
- Claim 与 Recover 触发阈值后进入冷却且无法绕过
- Recover 失败会触发指数退避
- 错误提示不泄露邮箱存在性/是否有邮件等元数据（与附录 A 一致）
